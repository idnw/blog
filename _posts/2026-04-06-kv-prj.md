# 实战项目方案：从内核驱动到Mini VMM

> 两个项目形成完整技术栈：驱动打基础 → VMM展虚拟化能力
> 预计总时间：6~8周（每天2~3小时）

---

## 项目一：virtio-echo — 一个虚拟virtio设备驱动（3~4周）

### 项目目标

在QEMU中实现一个自定义的virtio设备（echo device），并编写配套的Linux内核Guest驱动。Guest通过该设备发送数据，设备回显数据并附加时间戳。

**简历写法：** "设计并实现了自定义virtio设备及配套Linux内核驱动，涉及QEMU设备模型、virtqueue通信、中断处理、DMA映射及sysfs接口导出"

### 为什么选这个

1. virtio是面试高频考点，做过的人凤毛麟角
2. 同时涉及QEMU设备模型（C用户态）和Linux内核驱动（C内核态），一个项目展示两端能力
3. 涉及的内核子系统多：PCI、中断、DMA、sysfs、字符设备
4. 面试官一看就知道你真正动手写过代码

### 技术架构

```
┌────────────────────────────────────────┐
│              Guest Linux               │
│                                        │
│  用户程序 ←→ /dev/virtio-echo          │
│                  ↕                     │
│  virtio-echo内核驱动                    │
│    ├── virtqueue TX (发送数据)          │
│    ├── virtqueue RX (接收回显)          │
│    ├── 中断处理 (数据到达通知)           │
│    └── sysfs接口 (统计信息导出)         │
└──────────────┬─────────────────────────┘
               │ virtio (vring共享内存)
┌──────────────▼─────────────────────────┐
│              QEMU                       │
│                                        │
│  virtio-echo设备后端                    │
│    ├── QOM设备定义                      │
│    ├── virtqueue处理回调                │
│    ├── 数据处理 (echo + timestamp)      │
│    └── 中断注入                        │
└────────────────────────────────────────┘
```

### 分阶段实施

#### 阶段1：环境搭建与最小原型（第1周）

**目标：** Guest内核驱动能发现QEMU中的自定义virtio设备，probe成功。

```
任务清单：
□ 搭建开发环境
  - 编译自定义QEMU（基于stable分支，如v8.2）
  - 编译自定义Linux内核（建议6.1 LTS或6.6 LTS）
  - 准备rootfs（用busybox或buildroot最小系统）
  - 验证：自定义QEMU能启动自定义内核 + rootfs

□ QEMU侧：注册空设备
  - 在hw/virtio/下新建virtio-echo.c
  - 定义QOM TypeInfo，继承TYPE_VIRTIO_DEVICE
  - 分配自定义device ID（使用virtio保留的实验ID范围：0x30~0x3F）
  - 创建virtqueue（1个TX + 1个RX）
  - 编译验证：QEMU启动时设备出现在PCI总线上

□ Guest驱动侧：实现probe
  - 新建drivers/virtio/virtio_echo.c
  - 实现virtio_driver结构体，填写id_table匹配device ID
  - probe中：find_vqs()获取TX/RX virtqueue
  - insmod后dmesg看到probe成功日志
```

```c
typedef struct VirtIOEcho {
    VirtIODevice parent_obj;
    VirtQueue *tx_vq;
    VirtQueue *rx_vq;
    uint64_t tx_count;
    uint64_t rx_count;
} VirtIOEcho;

static void virtio_echo_handle_tx(VirtIODevice *vdev, VirtQueue *vq)
{
    // 阶段2实现
}

static void virtio_echo_device_realize(DeviceState *dev, Error **errp)
{
    VirtIODevice *vdev = VIRTIO_DEVICE(dev);
    VirtIOEcho *s = VIRTIO_ECHO(dev);

    virtio_init(vdev, VIRTIO_ID_ECHO, 0);  // 0 = config空间大小
    s->tx_vq = virtio_add_queue(vdev, 256, virtio_echo_handle_tx);
    s->rx_vq = virtio_add_queue(vdev, 256, NULL);
}
```

**关键代码骨架（Guest驱动侧）：**
```c
// drivers/virtio/virtio_echo.c
static struct virtio_device_id id_table[] = {
    { 0x30, VIRTIO_DEV_ANY_ID },  /* 匹配我们的echo设备 */
    { 0 },
};

static int virtecho_probe(struct virtio_device *vdev)
{
    struct virtqueue *vqs[2];
    vq_callback_t *callbacks[] = { virtecho_rx_done, NULL };
    const char *names[] = { "rx", "tx" };

    int err = virtio_find_vqs(vdev, 2, vqs, callbacks, names, NULL);
    if (err) return err;

    // 保存vqs到私有数据
    pr_info("virtio-echo: probe success, TX/RX queues ready\n");
    return 0;
}

static struct virtio_driver virtecho_driver = {
    .driver.name = "virtio-echo",
    .id_table    = id_table,
    .probe       = virtecho_probe,
    .remove      = virtecho_remove,
};
module_virtio_driver(virtecho_driver);
```

#### 阶段2：数据通路打通（第2周）

**目标：** 用户态程序写 /dev/virtio-echo → Guest驱动 → virtqueue → QEMU后端处理 → 回显 → Guest驱动 → 用户态读取

```
任务清单：
□ Guest驱动：实现字符设备接口
  - register_chrdev_region + cdev_add
  - file_operations: open/release/read/write/poll
  - write路径: copy_from_user → sg_init_one → virtqueue_add_outbuf → virtqueue_kick
  - read路径: 阻塞等待RX complete → copy_to_user
  - 中断回调: virtecho_rx_done → wake_up(等待队列)

□ QEMU后端：实现echo处理
  - virtio_echo_handle_tx:
    1. virtqueue_pop()取出请求
    2. 读取Guest数据
    3. 在数据前加上时间戳
    4. 将结果通过RX virtqueue返回
    5. virtio_notify()触发Guest中断

□ 测试验证
  - echo "hello" > /dev/virtio-echo
  - cat /dev/virtio-echo → 显示 "[timestamp] hello"
```

## 项目二：minivm — 从零实现Mini VMM（3~4周）

### 项目目标

用C语言直接调用KVM API（/dev/kvm），实现一个能启动真实Linux内核的最小虚拟机监控器。不依赖QEMU。

**简历写法：** "从零实现基于KVM API的轻量级VMM，支持Linux内核启动、EPT内存管理、串口设备模拟、virtio-console设备，深入理解VM Entry/Exit处理与虚拟化I/O路径"

### 为什么选这个

1. 市面上绝大多数人只用过QEMU/libvirt，真正调过KVM ioctl的极少
2. 直接展示你对VMX/EPT/VMCS的理解深度
3. 项目代码量可控（核心代码~2000行），但含金量极高
4. Firecracker、crosvm、cloud-hypervisor都是这个思路的生产级实现

### 技术架构

```
┌─────────────────────────────────────────┐
│              minivm进程                  │
│                                         │
│  main()                                 │
│   ├── kvm_init()        创建VM          │
│   ├── memory_init()     设置Guest内存    │
│   ├── load_kernel()     加载bzImage     │
│   ├── vcpu_init()       创建/配置vCPU   │
│   ├── devices_init()    初始化设备模拟    │
│   └── vcpu_run_loop()   VM Entry/Exit循环│
│        ├── KVM_RUN                      │
│        ├── handle_io()     PIO处理      │
│        ├── handle_mmio()   MMIO处理     │
│        └── handle_hlt()    HLT处理      │
│                                         │
│  设备模拟                                │
│   ├── serial_uart()     串口(0x3f8)     │
│   ├── i8042_stub()      键盘控制器桩     │
│   └── rtc_cmos()        RTC时钟         │
└──────────────┬──────────────────────────┘
               │ ioctl(/dev/kvm)
┌──────────────▼──────────────────────────┐
│           Linux KVM模块                  │
│  VMX/SVM硬件虚拟化                       │
└─────────────────────────────────────────┘
```

### 分阶段实施

#### 阶段1：最小可运行VM（第1周）

**目标：** 创建VM，加载一段16位汇编代码到Guest内存，运行并在串口输出"Hello from VM!"

```
任务清单：
□ KVM初始化
  - open("/dev/kvm") → KVM_CREATE_VM → KVM_CREATE_VCPU
  - KVM_SET_USER_MEMORY_REGION设置Guest物理内存
  - KVM_GET_VCPU_MMAP_SIZE → mmap vCPU的kvm_run结构

□ 编写16位Guest汇编代码
  - 在Guest内存0x0000处放置代码
  - 代码功能：向串口0x3f8写字符串
  - 设置vCPU的CS:IP指向代码起始地址

□ VM Exit处理循环
  - KVM_EXIT_IO: 检查port==0x3f8，打印字符到stdout
  - KVM_EXIT_HLT: 退出循环
  - 其他Exit: 打印信息并退出

□ 验证
  - 运行minivm → stdout显示 "Hello from VM!"
```

**核心代码框架：**
```c
// minivm.c - 阶段1核心
#include <linux/kvm.h>
#include <sys/ioctl.h>
#include <sys/mman.h>
#include <fcntl.h>

#define GUEST_MEM_SIZE (64 * 1024 * 1024)  /* 64MB */
#define SERIAL_PORT    0x3F8

struct vm {
    int kvm_fd;
    int vm_fd;
    int vcpu_fd;
    struct kvm_run *run;
    void *mem;          /* Guest物理内存（HVA） */
};

int vm_create(struct vm *vm)
{
    vm->kvm_fd = open("/dev/kvm", O_RDWR);
    // 检查KVM_API_VERSION

    vm->vm_fd = ioctl(vm->kvm_fd, KVM_CREATE_VM, 0);

    // 分配Guest内存
    vm->mem = mmap(NULL, GUEST_MEM_SIZE, PROT_READ | PROT_WRITE,
                   MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

    // 注册memory slot
    struct kvm_userspace_memory_region region = {
        .slot = 0,
        .guest_phys_addr = 0,
        .memory_size = GUEST_MEM_SIZE,
        .userspace_addr = (uint64_t)vm->mem,
    };
    ioctl(vm->vm_fd, KVM_SET_USER_MEMORY_REGION, &region);

    // 创建vCPU
    vm->vcpu_fd = ioctl(vm->vm_fd, KVM_CREATE_VCPU, 0);

    // mmap kvm_run
    int mmap_size = ioctl(vm->kvm_fd, KVM_GET_VCPU_MMAP_SIZE, 0);
    vm->run = mmap(NULL, mmap_size, PROT_READ | PROT_WRITE,
                   MAP_SHARED, vm->vcpu_fd, 0);
    return 0;
}

void vcpu_run_loop(struct vm *vm)
{
    while (1) {
        ioctl(vm->vcpu_fd, KVM_RUN, 0);

        switch (vm->run->exit_reason) {
        case KVM_EXIT_IO:
            if (vm->run->io.port == SERIAL_PORT &&
                vm->run->io.direction == KVM_EXIT_IO_OUT) {
                char *data = (char *)vm->run + vm->run->io.data_offset;
                write(STDOUT_FILENO, data, vm->run->io.size);
            }
            break;

        case KVM_EXIT_HLT:
            printf("\n[minivm] Guest halted.\n");
            return;

        case KVM_EXIT_MMIO:
            // 阶段3处理
            break;

        default:
            fprintf(stderr, "[minivm] Unhandled exit: %d\n",
                    vm->run->exit_reason);
            return;
        }
    }
}
```

#### 阶段2：启动真实Linux内核（第2周）

**目标：** 加载bzImage，设置boot参数，内核启动到串口输出内核日志（无需rootfs，panic在"unable to mount root fs"即可）

```
任务清单：
□ 实现bzImage加载器
  - 解析Linux boot protocol（Documentation/x86/boot.rst）
  - setup header在偏移0x01f1处
  - protected mode code加载到setup_header.code32_start指定的地址
  - 内核命令行加载到指定地址

□ 设置boot_params结构体
  - 在Guest内存0x10000处构造struct boot_params
  - 填写e820内存映射表
  - 设置命令行指针：console=ttyS0 earlyprintk=serial
  - 设置vid_mode等

□ 配置vCPU进入protected mode
  - KVM_SET_SREGS: 设置CR0(PE), GDT, 段寄存器为flat 32-bit模式
  - KVM_SET_REGS: RIP指向内核入口，RSI指向boot_params

□ 扩充串口模拟
  - 模拟16550 UART的基本寄存器（THR, LSR, IER）
  - LSR始终返回"TX empty"（简化实现）
  - 支持内核的earlycon/earlyprintk输出

□ 验证
  - 运行minivm vmlinuz → 看到内核启动日志
  - 日志显示到"Kernel panic - unable to mount root fs"为成功
```

#### 阶段3：完善设备模拟，启动到shell（第3周）

```
任务清单：
□ 准备initramfs
  - 用busybox制作最小initramfs
  - 或使用buildroot生成
  - 通过boot_params.initrd指定地址和大小

□ 补充必要设备模拟
  - i8042键盘控制器: 对端口0x60/0x64返回空响应（避免内核hang）
  - RTC(CMOS): 对端口0x70/0x71返回基本时间
  - PIC(8259): 基本的中断控制器桩（如果不用APIC的话）

□ 串口输入支持
  - 监听stdin → 当有输入时通过KVM_INTERRUPT注入串口中断
  - 实现串口RBR(接收)寄存器
  - 使Guest的getty/shell能通过串口交互

□ CPUID过滤
  - KVM_SET_CPUID2：设置Guest可见的CPUID信息
  - 过滤掉不需要暴露的特性

□ 验证
  - minivm vmlinuz initramfs.cpio.gz
  - 内核启动 → init → shell交互
  - 在shell中能执行命令（ls, cat /proc/cpuinfo等）
```

#### 阶段4：高级特性（第4周，面试重点加分项）

```
选做扩展（每个都是面试谈资）：

□ 多vCPU支持

  - 处理IPI（Inter-Processor Interrupt）
  - AP(Application Processor)启动流程：INIT-SIPI-SIPI

□ virtio-console实现
  - 实现virtio MMIO transport（比PCI简单）

□ 内存管理增强

□ 性能instrumentation
  - 统计VM Exit次数和原因分布

□ 简单的快照/恢复
```
