> 原则：每个项目都有真实用户/使用场景，不是"为了学而学"
> 面试官看到会觉得"这个人解决过真实问题"

## 项目一：VM性能诊断工具（推荐度最高）

### 解决什么问题
生产环境中KVM虚拟机性能抖动是运维最头疼的问题之一。现有工具要么太重（perf全量采集），要么信息不全（只看Guest或只看Host）。缺一个**轻量级、实时、Host+Guest联动**的诊断工具。

### 做什么
用eBPF/bpftrace开发一个命令行工具集（类似BCC tools风格），专门诊断KVM虚拟机性能问题：

```
kvmdiag exit-stats <pid>       # 实时统计目标VM的VM Exit原因分布和延迟
kvmdiag steal-time <pid>       # 追踪vCPU被Host抢占的时间（steal time分析）
kvmdiag mmio-trace <pid>       # 记录MMIO访问热点（哪个设备导致最多Exit）
kvmdiag ept-fault <pid>        # EPT violation统计与分析
kvmdiag irq-latency <pid>     # 虚拟中断从注入到Guest处理的延迟
kvmdiag vcpu-preempt <pid>    # 检测Lock Holder Preemption问题
kvmdiag overview <pid>         # 一键综合报告
```

### 技术要点
- eBPF程序hook KVM tracepoint（`kvm:kvm_exit`, `kvm:kvm_entry`, `kvm:kvm_mmio`等）
- 用BPF map做实时聚合，低开销
- 用户态用Python/Go展示（直方图、top-N、时间线）
- 结合`/proc/pid/schedstat`分析vCPU调度质量

### 为什么有说服力
- **真实痛点**：所有KVM用户都遇到过虚拟机卡顿排查困难的问题
- **可以开源**：BCC/bpftrace社区对KVM相关工具需求大，有可能被合入上游
- **展示深度**：需要理解KVM内部机制才能写出有意义的诊断逻辑
- **简历写法**："开发了基于eBPF的KVM虚拟机性能诊断工具集，实时追踪VM Exit、vCPU调度延迟、中断注入时延等指标，协助定位了线上数据库虚拟机延迟抖动问题"

## 项目二：安全容器运行时（secvm）

### 解决什么问题
Kata Containers/gVisor解决的问题：容器隔离不够强，但传统VM太重。做一个**极简的、用KVM给每个容器加一层硬件隔离**的运行时。

### 做什么
一个兼容OCI标准的容器运行时，底层用KVM而不是namespace做隔离：

```bash
# 像Docker一样使用，但底层是VM隔离
secvm run --memory 128m --cpus 1 alpine:latest /bin/sh

# 兼容containerd，可以被K8s调度
# crictl pull docker.io/library/nginx
# crictl run pod.json container.json
```

### 核心模块
```
secvm-runtime (OCI runtime, 类似runc的角色)
├── vm.c          — KVM API创建/管理轻量VM
├── kernel/       — 定制的轻量Guest内核(裁剪到5秒内启动)
├── agent/        — Guest内运行的agent进程(管理容器进程生命周期)
├── drive.c       — 将容器rootfs作为virtio-blk挂载到Guest
├── network.c     — tap+bridge网络，一个VM一个tap
├── jailer.c      — seccomp + cgroup限制VMM进程本身
└── shimv2/       — containerd shim接口(Go, 可选)
```

### 为什么有说服力
- **直接对标业界方案**：Kata Containers(蚂蚁/Intel)、Firecracker(AWS)、gVisor(Google) — 这是大厂在做的事
- **全栈能力展示**：KVM API + 内核裁剪 + 设备模拟 + 容器生态对接
- **面试谈资极多**：安全隔离模型对比、启动时间优化、内存开销控制
- **可以真正用起来**：替代docker run做安全隔离测试

## 项目三：内核热补丁框架（livepatch工具）

### 解决什么问题
生产服务器内核漏洞修复需要重启，但很多场景不能停机（数据库、关键业务）。Linux有livepatch机制，但**制作补丁的工具链很难用**，缺少一个简单的"一键生成热补丁"的工具。

### 做什么
```bash
# 输入：原始内核源码 + 补丁diff
# 输出：可以直接insmod的livepatch .ko模块

klpgen --kernel /usr/src/linux-6.1.0 \
       --patch fixes/cve-2024-xxxx.patch \
       --output cve-2024-xxxx-livepatch.ko

# 一键应用
insmod cve-2024-xxxx-livepatch.ko
cat /sys/kernel/livepatch/cve_2024_xxxx/enabled  # 1 = 生效
```

### 核心实现
```
klpgen/
├── diff_parser.c      — 解析patch，提取修改了哪些函数
├── symbol_resolve.c   — 通过kallsyms/vmlinux找到原函数地址
├── klp_gen.c          — 生成livepatch模块的C代码框架
├── ftrace_hook.c      — 基于ftrace的函数替换（livepatch底层机制）
├── consistency.c      — 一致性模型检查（确保所有线程都不在被替换函数中）
└── build.sh           — 编译生成.ko
```

### 为什么有说服力
- **运维刚需**：大公司内核团队的核心工作之一就是热补丁
- **内核深度**：需要理解ftrace、kallsyms、模块加载、栈回溯
- **安全相关**：CVE修复场景，面试官会追问安全相关的细节
- **华为/阿里/腾讯内核团队特别认可**：这些公司都有自己的热补丁方案(kpatch/livepatch)

## 项目四：设备直通性能测试与对比平台

### 解决什么问题
虚拟化I/O方案选型（virtio vs SR-IOV vs VFIO直通 vs vDPA）缺乏标准化的对比数据，每次选型都要从头测一遍。

### 做什么
一个自动化的虚拟化I/O性能测试框架：

```bash
vio-bench --scenarios all --device nvme0n1 --report report.html

# 自动执行：
# 1. 裸机基线测试 (fio/iperf3)
# 2. virtio-blk + QEMU用户态后端
# 3. virtio-blk + vhost内核后端
# 4. virtio-blk + io_uring后端
# 5. VFIO NVMe直通
# 6. SR-IOV VF直通 (如果网卡支持)
# 7. vDPA (如果支持)
#
# 生成对比报告：IOPS、带宽、延迟分布(P50/P99/P999)、CPU开销
```

### 核心组件
```
vio-bench/
├── host_setup/
│   ├── vfio_bind.sh      — 自动绑定设备到vfio-pci
│   ├── sriov_setup.sh    — 创建VF并配置
│   └── hugepage_setup.sh — 配置大页内存
├── vm_templates/
│   ├── virtio_blk.xml    — libvirt XML模板
│   ├── vfio_passthrough.xml
│   └── sriov_vf.xml
├── guest_bench/
│   ├── fio_profiles/     — 标准化fio测试配置
│   └── iperf_profiles/
├── analysis/
│   ├── collect.py        — 收集Host+Guest指标(VM Exit, CPU%, 中断数)
│   ├── compare.py        — 多方案对比分析
│   └── report.py         — 生成HTML报告(图表)
└── vmexitmon/
    └── bpf_kvm_exit.py   — 测试期间eBPF采集VM Exit数据(与项目一联动)
```

### 为什么有说服力
- **选型决策依据**：任何用KVM的公司都需要这类数据
- **可以写成技术文章**：对比数据本身就是硬核内容，发到博客/知乎影响力大
- **覆盖面广**：VFIO、SR-IOV、vhost、IOMMU一个不落
- **可持续维护**：每次内核/QEMU升级都可以跑一轮，积累历史数据

## 项目五：给Linux内核提交真实补丁

### 为什么这是最有说服力的"项目"
没有任何项目比**"我的代码被合入Linux内核主线"**更有说服力。这不是一个大项目，而是一个持续的行为。

### 怎么做

**入门路径（真实可行）：**

```
1. 从cleanup/fix开始
   - 跑 scripts/checkpatch.pl 修代码风格（但别只提这类补丁，太初级）
   - 跑 sparse/smatch/coccinelle 找真实bug
   - 例: coccinelle脚本检测缺少of_node_put()的DT驱动

2. 找一个你用过的驱动/子系统
   - 看它的TODO文件或MAINTAINERS
   - staging驱动(drivers/staging/)有很多明确标注需要做的事
   - 例: 将一个staging驱动从旧API迁移到新API

3. 修真实Bug
   - 看 bugzilla.kernel.org
   - 看邮件列表(lkml/kvm@vger)中报告但无人修的bug
   - syzkaller报告的bug(很多小bug没人修)

4. KVM相关贡献（高价值）
   - KVM自测试改进(tools/testing/selftests/kvm/)
   - KVM文档补充
   - 复现并修复kvm邮件列表上报告的问题
```

**简历写法**："向Linux内核主线贡献了X个补丁，涉及KVM/驱动/XX子系统，被Reviewed-by: [maintainer name]"

### 为什么这是最终武器
- **不可伪造**：git log里有你的名字，面试官可以直接查
- **证明协作能力**：经历过邮件列表review流程
- **技术认可**：maintainer的Reviewed-by/Acked-by就是最好的背书

## 我的建议：组合打法

```
立刻开始（边面试边做）：
  项目一（eBPF诊断工具）— 2~3周出成果，最快体现能力

中期（1~2个月）：
  项目五（内核补丁）— 提交2~3个有质量的补丁

如果时间充裕（2~3个月）：
  项目二（安全容器运行时）— 最完整的能力展示
```

项目一入门快、实用性强，面试时能直接demo。项目五是长期投资，但回报最大。两者可以同时推进。
